> **PROPOSAL ONLY — no code in this document.** Builds a launchable macOS app

## 1. Context

- **Current status:** Gates 1–13 merged. A deterministic, lock-free engine plays
  UASF styles; the Gate 12 performer modules are wired into `StylePlayer` via the
  Gate 13 `PerformerAdapter`. CoreMIDI output exists (Gate 9). 46 test binaries /
  576 assertions, CI green. No GUI yet.
- **This gate IS:** a thin macOS app *shell* — windows, panels, buttons, chord
  input, status/diagnostics — that drives the engine through the existing
  control/adapter seam.
- **This gate IS NOT:** a DAW, a synth, an audio/DSP layer, a plugin host, or a
  hardware integration. No sound is generated by the app; it sends MIDI/control
  intent to the engine.

## 2. Strategic Goal

A **playable software-instrument shell**: the UI is a pure client of the engine.
It visualizes state and emits performer intent; it never owns arranger logic.
This is the first end-user-facing surface of the KORG-first platform.

## 3. Platform Decision

- **macOS first** (the engine + CoreMIDI already target macOS).
- **iPadOS later** via the same JUCE codebase (no rewrite).

## 4. Technology Constraints

- **Allowed UI framework:** JUCE (primary), or minimal native macOS (AppKit/SwiftUI)
  only if strictly necessary for a shell concern.
- **Forbidden:** Electron, any webview/web stack, Flutter, React Native, Qt.
- **Reasoning:** the engine is C++; a C++ UI toolkit means zero binding overhead,
  one language, and a clean realtime-thread model. Web/cross-runtime stacks add
  IPC, GC pauses, and nondeterminism that conflict with the realtime contract.

## 5. Hard Architectural Rules

- **Strict engine/UI separation:** the UI sends *intent* only. It does NOT own
  arranger logic, does NOT quantize, does NOT schedule, does NOT compute timing.
  All of that stays in the engine.
- **No blocking UI→engine calls:** communication is async-safe via a lock-free
  event queue (UI thread ↔ engine thread). No UI-thread blocking, no mutex
  contention, no synchronous render callbacks into the engine.
- **Determinism preserved:** the UI introduces no nondeterminism, never bypasses
  the scheduler, never manipulates timing. Every report emits `deterministic:
  true` + `hardware_validated: false`.

## 6. Scope (10 items)

1. **macOS App Shell** — launchable window hosting the panels below.
2. **Engine Session Lifecycle** — boot / shutdown / reset of the engine from the
   app (clean, repeatable, panic-safe).
3. **Realtime Transport UI** — Start/Stop/Sync/Intro/Ending/Fill/Break/
   Variation A–D/Panic, each emitting a `ControlEvent`.
4. **Chord Input UI** — virtual chord pad + computer-keyboard entry + current
   chord display, routed through the chord-scan modes.
5. **Realtime Status Visualization** — current section, bar/beat, playing state,
   active variation, note activity.
6. **MIDI Device Layer** — enumerate CoreMIDI destinations, select output,
   hot-plug safe.
7. **Internal Event Console** — scrolling log of control events + engine
   responses (debug/observability).
8. **Latency Telemetry** — live control→dispatch latency display, backed by the
   Gate 12/13 latency reporter.
9. **Style Browser (Minimal)** — list/load `.uasf` styles from a folder.
10. **Snapshot State System** — capture/restore the performer+session state
    (variation, chord, split, sync, selected style/device).

## 7. Suggested Structure

`apps/macos/` (JUCE app), `src/session/` (lifecycle + snapshot), `src/control/`
(async UI↔engine bridge + queue), `src/ui/` (UI-toolkit-agnostic ViewModels),
`src/midi/` (CoreMIDI device manager), `tests/integration/` + `tests/ui/`
(headless). Detailed layout in §13.

## 8. Testing Requirements

- Boot / shutdown (repeatable, no leaks/hangs).
- Panic during playback (zero hanging notes, re-arm).
- Rapid fill spam (collapse, no double-fire).
- MIDI device reconnect / hot-plug.
- Snapshot capture → restore round-trip.
- Sync-start race (concurrent).
- UI event flood through the bridge (no loss/reorder, bounded).
- Deterministic replay (same control sequence → same state).
- **All headless-safe + synthetic-only** — no GUI, no real device, no network in
  CI tests.

## 9. Deliverables

- A launchable macOS prototype.
- Screenshots (manual or programmatic).
- Latency + deterministic-replay reports (`deterministic: true` +
  `hardware_validated: false`).
- Updated `GATE_14_HANDOFF.md` + `HANDOFF_MASTER.md`.
- CI green (engine + headless tests; the GUI build may be CI-optional).

## 10. Forbidden

## 11. Success Criteria

- The app launches and drives the engine: transport + chord input produce real

- UI is a pure client (no arranger logic / timing in the UI).
- All cross-thread comms are async + lock-free (no UI-thread blocking).
- Snapshot capture/restore works.
- The 46 baseline binaries / 576 assertions stay green; new headless tests pass.
- Reports carry `deterministic: true` + `hardware_validated: false`.

## 12. JUCE choice rationale

- **Cross-platform macOS → iPadOS** from one C++ codebase (no rewrite for iPad).
- **Native CoreMIDI** integration (already used in Gate 9) fits JUCE's model.
- **Lock-free message-thread / audio-thread split** is first-class in JUCE,
  matching the engine's realtime contract.
- **No web stack / no platform GUI lock-in** — avoids Electron/webview
  nondeterminism and IPC.
- **C++ engine + C++ JUCE = zero binding overhead**, one toolchain, one language.

## 15. CI strategy

- `BUILD_MACOS_APP=OFF` by default → CI builds the engine + all headless tests
  (session, bridge, viewmodels, midi manager, snapshot, latency) via the existing
  `scripts/run_tests.sh`; **no workflow change**.
- The JUCE GUI target builds locally (and can be enabled in a later CI lane).
- Screenshots are **manual** (PTH runs locally) initially; an `xcodebuild`/UI-test
  runner is a Gate 14+ follow-up.
- No CoreMIDI device calls in tests — the MIDI manager is tested via a synthetic
  enumeration seam.
